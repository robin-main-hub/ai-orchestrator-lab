# 외부 모델 검토 계획

## 목표

AI Orchestrator Lab의 설계를 Claude, GPT, Gemini, 로컬 모델, 코딩 특화 모델에게 각각 검토시켜 blind spot을 줄인다. 검토는 단순 칭찬이나 아이디어 수집이 아니라, 실제 구현 전에 위험한 구조, 빠진 요구사항, 과도한 복잡도, 보안 문제, 사용자 경험 문제를 찾는 데 집중한다.

## 검토 원칙

- 모든 모델에게 같은 기준 문서를 제공한다.
- 모델별로 다른 역할을 준다.
- 답변은 `docs/review-board.md`에 모아 비교한다.
- 좋은 아이디어도 바로 채택하지 않고, 제품 원칙과 구현 가능성에 맞춰 선별한다.
- 모델이 제안한 기능은 반드시 우선순위, 비용, 위험, 대체안을 함께 평가한다.

## 모델별 역할

| 검토자 | 역할 | 중점 |
| --- | --- | --- |
| Claude 계열 | 제품/UX/에이전트 설계 리뷰어 | 흐름의 자연스러움, 사용자의 작업 감각, soul/memory의 위험 |
| GPT 계열 | 시스템 아키텍처 리뷰어 | 프로토콜, 이벤트 모델, 서버/로컬 폴백, 구현 순서 |
| Gemini 계열 | 넓은 제품 전략 리뷰어 | 빠진 기능, 모바일/멀티모달/장기 확장성 |
| 로컬 모델 | Offline 현실성 리뷰어 | 로컬 폴백, 작은 모델에서의 동작, 비용 절감 |
| 코딩 특화 모델 | 구현 가능성 리뷰어 | 모노레포, API 경계, 테스트, 패키지 분리 |

## 공통 검토 질문

1. 이 설계에서 가장 먼저 깨질 가능성이 높은 부분은 무엇인가?
2. 사용자가 실제로 매일 쓰려면 어떤 흐름이 더 단순해야 하는가?
3. 프로바이더/API 키/리셀러 호환 설계에서 빠진 위험은 무엇인가?
4. DGX 서버와 로컬 폴백 경계가 충분히 명확한가?
5. 대화형 작업 모드와 토론 모드는 같은 제품 안에서 자연스럽게 공존하는가?
6. `soul.md`는 장기적으로 도움이 되는가, 아니면 경직성과 비용을 키우는가?
7. Memento 메모리, Obsidian 백업, Notion 동기화의 역할 분리가 적절한가?
8. 외부 인입/OpenClaw 브리지는 어떤 보안/권한 문제가 생길 수 있는가?
9. 구현 순서에서 바꿔야 할 것이 있는가?
10. 반드시 빼야 하거나 나중으로 미뤄야 할 기능은 무엇인가?

## 공통 요청 프롬프트

```text
너는 AI Orchestrator Lab의 외부 설계 리뷰어다.

이 프로젝트는 맥북 데스크톱 앱과 dgx-02 서버를 함께 사용하는 AI 오케스트레이터다. 사용자는 토론 모드, 1:1 대화형 작업 모드, 코딩 전달, 멀티 프로바이더, 로컬 폴백, Memento 스타일 메모리, Obsidian/Notion 백업, 외부 인입/OpenClaw 브리지를 모두 원한다.

아래 문서를 읽고 칭찬보다 결함과 위험을 우선해서 리뷰해라.

반드시 다음 형식으로 답하라.

1. 가장 큰 위험 5개
2. 빠진 요구사항 또는 모호한 요구사항
3. 구현 순서에 대한 수정 제안
4. 보안/권한/데이터 보존 리스크
5. UX가 복잡해질 지점
6. 반드시 유지해야 할 좋은 설계
7. 다음 구현 단계에서 확인해야 할 질문
8. 네가 이 프로젝트의 설계자라면 바꿀 것

가능하면 구체적인 파일명과 섹션명을 언급해라.
막연한 조언보다 실행 가능한 제안을 줘라.
```

## 결과 수집 방식

각 모델의 답변은 `docs/review-board.md`에 다음 기준으로 정리한다.

- 공통 지적
- 모델별 고유 지적
- 바로 반영할 것
- 보류할 것
- 반영하지 않을 것
- 새로 생긴 질문

## 채택 기준

외부 모델의 제안은 다음 조건을 통과해야 한다.

- 제품 목표와 맞는가?
- 구현 순서를 심하게 꼬지 않는가?
- 보안과 권한 모델을 약하게 만들지 않는가?
- 로컬 폴백 철학을 해치지 않는가?
- 사용자가 실제로 매일 쓸 때 더 편해지는가?
