# UI 방향

## 핵심 톤

프론트엔드는 장난감 같은 채팅방이 아니라, 여러 AI를 지휘하는 작업실이어야 한다. 어두운 패널, 명확한 구획, 모델별 상태, 실행 로그, 토론 흐름이 한눈에 들어와야 한다.

다만 토론이 꺼진 상태에서는 사용자가 OpenClaw나 Claude와 대화하듯 자연스럽게 일할 수 있어야 한다. 이때도 앱은 단순 채팅방이 아니라 기록, 메모리, 백업, 코딩 전달이 살아 있는 Conversation Workbench로 동작한다.

## 첫 화면

첫 화면은 랜딩 페이지가 아니라 실제 오케스트레이터다.

- 좌측: 세션/프로젝트/프로바이더/채널 네비게이션
- 중앙: 오케스트레이터 작업판
- 우측: 모델/에이전트/메모리 상태와 상세 패널
- 하단: 터미널과 실행 로그
- 상단: Status Hub

중앙 작업판은 토론 토글에 따라 바뀐다.

- 토론 Off: Conversation Workbench
- 토론 On: Debate Table

기본 진입은 Conversation Workbench다. Debate Table은 사용자가 대화 중 `토론으로 전환`을 선택하거나 오케스트레이터가 승격을 제안할 때 열린다.

## Status Hub

Status Hub는 장식용 요소가 아니라 클릭 가능한 상태 요약 컨트롤이다. 상단에 작게 배치하고, 클릭하면 상세 패널을 연다.

Status Hub가 요약할 항목:

- DGX Online/Degraded/Offline/Syncing
- 로컬 모델 사용 가능 여부
- 현재 프로바이더와 모델
- 예상 비용/토큰 예산
- Permission pending 개수
- Redaction 적용 여부
- Obsidian/Notion 백업 상태
- Memento 동기화 상태

## 주요 화면

### Orchestrator Board

작업 목표를 입력하고, 모드와 에이전트 구성을 선택한다. 대화, 토론, 코딩, 리뷰, 검증이 모두 여기서 시작된다.

### Conversation Workbench

토론 기능을 끈 상태에서 AI와 1:1로 대화하며 일하는 화면이다. 사용자는 OpenClaw, Claude, Codex, 로컬 모델, 커스텀 에이전트 중 하나를 대화 상대로 고른다.

Conversation Workbench에는 다음 액션이 있어야 한다.

- 토론으로 전환
- 코딩 패킷 만들기
- 실행 슬롯으로 보내기
- 메모리에 저장
- Obsidian/Notion 백업 상태 보기
- Telegram에서 이어받기

### Debate Table

모델별 발언을 나란히 보여준다. 각 발언은 합의, 반대, 근거, 리스크, 코딩 영향으로 태깅된다.

### Coding Handoff

대화 또는 토론 결과를 구현 가능한 패킷으로 바꿔 보여준다. 사용자는 이 패킷을 특정 CLI 에이전트나 터미널 슬롯에 전달한다.

### Provider Profiles

여러 API 키와 base URL을 동시에 등록한다. 붙여넣은 환경변수에서 자동 추출하고, 모델 목록을 불러와 선택한다.

### Memory Inspector

Memento 스타일 기억을 검색하고 관리한다. 기억이 어떤 결정에 쓰였는지 추적한다.

## 디자인 규칙

- 카드 남발보다 작업판/패널 중심으로 구성한다.
- 모델 상태는 색과 아이콘으로 빠르게 구분한다.
- 버튼에는 가능한 한 아이콘을 붙인다.
- 실행 중인 작업은 스트리밍 상태가 보여야 한다.
- 텍스트 설명보다 실제 조작 가능한 컨트롤을 우선한다.
- 서버 Offline 상태에서는 비활성 기능을 숨기지 말고 왜 제한됐는지 표시한다.
- 대화형 모드에서도 실행/파일 변경/터미널 명령은 권한 정책을 따른다.
- 상태 정보는 여러 패널에 흩뿌리지 않고 Status Hub에서 요약한 뒤 필요할 때 상세화한다.
