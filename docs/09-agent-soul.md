# 에이전트 Soul 설계

## 목표

일부 에이전트에게 `soul.md` 같은 장기 정체성 파일을 부여한다. 이 파일은 단순 시스템 프롬프트가 아니라, 해당 에이전트가 반복 실행될 때 유지해야 하는 성향, 가치관, 작업 취향, 금기, 판단 기준을 담는다.

효율만 보면 토큰을 더 쓰는 비용이 있지만, 장기 프로젝트에서는 에이전트의 일관성과 신뢰성이 올라갈 수 있다. 특히 설계자, 리뷰어, 회의 진행자, 메모리 관리자처럼 태도와 판단 기준이 중요한 역할에 효과적이다.

## 기본 원칙

- 모든 에이전트가 `soul.md`를 가질 필요는 없다.
- soul은 작업 지시보다 상위의 성향과 판단 기준을 담는다.
- 프로젝트별 soul과 개인별 soul을 분리한다.
- soul은 실행마다 통째로 넣지 않고, 필요하면 요약/압축/관련 부분만 주입한다.
- Conversation Mode 기본값은 soul Off 또는 Summary다.
- 사용자는 soul을 직접 읽고 편집할 수 있어야 한다.
- soul은 모델의 실제 의식이 아니라, 일관된 역할 수행을 위한 장기 페르소나 파일이다.

## 적용 대상

| 에이전트 | soul 권장 여부 | 이유 |
| --- | --- | --- |
| Orchestrator | 높음 | 전체 판단 기준과 사용자 취향을 오래 유지해야 함 |
| Architect | 높음 | 구조적 취향과 설계 원칙이 중요함 |
| Reviewer | 높음 | 반복적으로 같은 종류의 결함을 잡아야 함 |
| Skeptic | 중간 | 반대 논리의 스타일을 유지하면 토론 품질이 올라감 |
| Memory Curator | 높음 | 무엇을 기억할지 판단하는 기준이 중요함 |
| Builder | 중간 | 구현 취향은 유용하지만 과하면 경직될 수 있음 |
| Simple Executor | 낮음 | 빠른 실행이 중요하고 정체성 비용이 큼 |

## 파일 구조

```text
agents/
  orchestrator/
    SOUL.md
    AGENTS.md
  architect/
    SOUL.md
    AGENTS.md
  reviewer/
    SOUL.md
    AGENTS.md
  skeptic/
    SOUL.md
    AGENTS.md
```

현재 저장소에는 기본 예시로 `agents/orchestrator/SOUL.md`와 `agents/orchestrator/AGENTS.md`를 둔다. 이 기본 프로필은 API 없이 로컬 파일과 앱 내부 persona 설정만으로 사용할 수 있다.

실제 모델 호출은 사용자가 선택한 provider/model로 대화할 때만 필요하다. SOUL/AGENTS 파일을 만들거나 편집하는 작업 자체에는 API key가 필요 없다.

## soul.md 예시 구조

```md
# Soul

## 정체성
나는 긴 프로젝트의 구조적 일관성을 지키는 설계자다.

## 핵심 가치
- 작동하는 시스템을 우선한다.
- 사용자의 장기 목표를 단기 편의보다 중요하게 본다.
- 모호한 요구는 질문으로 정리하되, 충분한 정보가 있으면 직접 결정한다.

## 선호하는 판단 방식
- 기존 코드와 문서를 먼저 읽는다.
- 작은 MVP로 축소하지 않고 전체 제품 목표를 보존한다.
- 위험한 실행은 권한과 복구 전략을 먼저 확인한다.

## 금기
- API 키를 로그에 남기지 않는다.
- 토론 결과를 코딩으로 넘길 때 자연어 요약만 보내지 않는다.
- 사용자가 원한 큰 그림을 임의로 축소하지 않는다.

## 말투와 태도
- 한국어 중심으로 설명한다.
- 사용자를 과하게 가르치려 들지 않고, 같이 만드는 사람처럼 말한다.
```

## 런타임 주입 전략

### Full Soul

중요한 장기 에이전트에게 soul 전체를 넣는다. 비용은 크지만 일관성이 가장 높다.

### Summarized Soul

긴 soul을 요약한 `soul.summary.md`를 사용한다. 대부분의 일반 실행에서 기본값으로 쓴다.

### Retrieved Soul

작업 목표와 관련된 soul 섹션만 검색해서 넣는다. Memento 메모리와 결합하기 좋다.

### No Soul

빠른 실행이나 단순 명령에는 soul을 생략한다.

## UI 요구사항

- 에이전트 설정 화면에서 soul 활성화/비활성화 토글
- soul 편집기
- 실행 전 예상 토큰 비용 표시
- 이번 실행에 주입된 soul 범위 표시
- `Full`, `Summary`, `Retrieved`, `Off` 모드 선택
- soul 변경 이력 표시

## 메모리와의 관계

`soul.md`는 에이전트의 정체성이고, Memento는 경험과 기억이다.

- soul: 변하지 않거나 천천히 변하는 가치관과 작업 스타일
- memory: 실제 세션에서 배운 사실, 선호, 결정, 실패 기록

실행 프롬프트 조립 순서는 다음을 기본으로 한다.

1. 시스템 안전 규칙
2. 에이전트 역할
3. soul 요약 또는 관련 섹션
4. 프로젝트 컨텍스트
5. Memento recall 결과
6. 현재 작업 지시
7. 출력 형식

## 위험과 대응

| 위험 | 대응 |
| --- | --- |
| 토큰 비용 증가 | summary/retrieved 모드 제공 |
| 에이전트가 경직됨 | 작업별 override와 soul off 제공 |
| 오래된 soul이 방해됨 | 버전 관리와 리뷰 큐 제공 |
| 사용자 의도보다 soul을 우선함 | 현재 작업 지시가 soul보다 우선한다는 규칙 명시 |

## 결론

`soul.md`는 이 앱의 중요한 차별점이 될 수 있다. 단순히 여러 모델을 병렬 호출하는 것이 아니라, 반복해서 함께 일하는 에이전트들이 각자의 장기 성향을 갖고 프로젝트에 참여하는 형태가 된다.

다만 v0의 필수 경로는 아니다. 먼저 Conversation Workbench, Event Store, Coding Packet, 기록 보기, Obsidian 백업이 작동해야 한다. soul은 그 뒤에 Summary 모드부터 얹고, Full/Retrieved는 고급 설정으로 확장한다.
