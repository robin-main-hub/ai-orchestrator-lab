# 63 — 코딩 워크벤치 모델 드롭다운 + 우측 버블 + 오류 진단 (기능 개편 F#1)

코딩 탭(CodingWorkbench)의 세 가지 불편: (1) 모델을 텍스트로 직접 입력, (2) 내가 쓴
메시지가 좌측에 붙어 채팅 같지 않음, (3) "Failed to fetch" 오류가 불친절.

## 한 일 (회귀 0)

- **모델 탑다운 선택**: 모델 `<input>`(텍스트) → 프로바이더 기반 `<select>`. 선택한
  프로바이더의 `modelCatalog[providerProfileId]` 모델만 옵션으로. 카탈로그가 비어도
  defaultModel과 현재 선택 모델은 옵션으로 남겨 선택을 잃지 않는다. 프로바이더 미선택
  시 "프로바이더 먼저 선택"으로 비활성. App이 `modelCatalog`를 주입.
- **사용자 메시지 우측 버블**: `.coding-message--user`를 우측 정렬 + 시안 버블
  (`14px 14px 4px 14px` 말풍선). 에이전트 응답은 좌측 풀폭 유지(코드/툴 카드용).
- **오류 진단·친절화**: LLM 호출은 오케스트레이터 서버(:4317)의 `/provider-completions
  (/stream)`로 가므로 "Failed to fetch"는 프로바이더가 아니라 **그 서버에 못 닿은 것**.
  `humanizeCodingError`(순수·테스트)가 네트워크 류 오류를 "오케스트레이터 서버(:4317)에
  연결할 수 없습니다 — 서버 실행·Tailscale 확인"으로 바꾸고, 원문은 title로 보존.

## "Failed to fetch" 진단 결과 (정직하게)

이건 UI 코드 버그가 아니라 **환경 문제**다. 코딩 워크벤치는 LLM을 직접 부르지 않고
오케스트레이터 서버(`127.0.0.1:4317` 또는 `dgx-02:4317`)를 거치는데, 그 서버가 실행
중이 아니거나 Tailscale로 닿지 않으면 fetch가 거부돼 "Failed to fetch"가 난다. 해결은
서버를 띄우거나 dgx-02 연결을 확인하는 것 — 코드로 고칠 수 없어, 대신 오류 문구를
원인을 가리키게 바꿨다.

## 원칙

- **탑다운 선택**: 외워서 타이핑 대신 프로바이더→모델 드롭다운.
- **거짓말하지 않는다**: 오류의 진짜 원인(서버 연결)을 가리키고, 원문은 보존.
- **회귀 0**: 카탈로그 폴백으로 기존 모델 선택 유지. desktop 1126 그린(+4), 빌드 통과.
